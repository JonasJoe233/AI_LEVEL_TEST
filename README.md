# AITI · AI Type Indicator

> 测你的 AI 等级 —— 一个病毒式 AI 使用水平测试工具

**在线体验**：[https://jonasjoe233.github.io/AI_LEVEL_TEST/](https://jonasjoe233.github.io/AI_LEVEL_TEST/)

---

## 项目思路

### 灵感来源

灵感来自卡兹克公众号文章《观察了三年，我把所有人用AI的水平分成了10个等级》。文章提出了一个有趣的框架：AI 使用水平不是单一维度，而是四个独立进阶的维度组合。

我们想把这个框架变成一个**可传播的测试工具**，类似 MBTI 人格测试，但测的是「AI 使用水平」。

### 产品定位

- **目标受众**：普通 AI 用户（非专业开发者）
- **传播场景**：微信 / 小红书 / 朋友圈
- **核心价值**：让人知道自己在哪里、笑一笑、顺手分享
- **对标产品**：SBTI（MBTI 恶搞版饮酒人格测试）

### 命名

**AITI** = **AI Type Indicator**

和 MBTI（Myers-Briggs Type Indicator）一眼同族，看到就知道在蹭什么。

---

## 具体做法

### 四维模型

原文框架直接沿用，四维度不并行，各自独立进阶：

| 维度 | 代号 | 测什么 |
|------|------|--------|
| 可控性 | C | 能不能驯服 AI，约束它精准产出 |
| 广度 | B | 把 AI 用到多宽的领域 |
| 形态 | F | 停留在 ChatBot 还是进阶到 Agent |
| 角色 | R | 消费者还是创造者 |

### 题目设计

- **20 道情景选择题**，每维度各 5 题
- **A/B/C 三选一**，计分简单（1/2/3 分）
- **交叉排列**：C, B, F, R 循环，避免连续同类题感
- **每维度离散化**：L (≤8) / M (9-11) / H (≥12)

### 等级系统

总分 20-60 分，映射到 Lv.0-10：

| 等级 | 总分区间 | 稀有度 | 预估占比 |
|------|---------|--------|---------|
| Lv.0-2 | <28 | 普通 | ~60% |
| Lv.3-4 | 29-35 | 稀有 | ~25% |
| Lv.5-6 | 36-43 | 精英 | ~10% |
| Lv.7-8 | 44-51 | 传说 | ~4% |
| Lv.9-10 | ≥52 | 神话 / ??? | ~1% |

### 人设匹配

**16 个正常人设**：基于四维度 H/M/L 组合，如：
- `LURK` 潜伏者（L,L,L,L）→ Lv.0-1
- `SOLO` 一人军团（H,H,H,H）→ Lv.9-10
- `CODE` 工程师（H,H,H,M）→ Lv.8

**匹配算法**：与模板比对，完全匹配 +3 分，相差 1 级 +1 分，取最高分。

**3 个彩蛋人设**（异常检测风格）：
- `GOD` AI本尊：全部选 C → 你是在用 AI，还是你就是 AI？
- `CAVE` 数字难民：全部极低 → 2026 年 AI 时代的抵抗者
- `HYPE` 嘴强王者：R 高但 C/F 低 → 知道很多，实操落差大

## 创新点

### 1. 彩蛋机制

不沿用 SBTI 的「饮酒触发」，改为「AI 异常检测风格」：

> 测试本身表现得像 AI —— 检测到异常答题模式，介入报告。

这与「AI 水平测试」主题高度契合，更有传播爆点。

### 2. 稀有度系统

类似游戏装备稀有度：普通 → 稀有 → 精英 → 传说 → 神话 → ???

激发收集欲和炫耀欲，促进分享传播。

### 3. 纯前端实现

- **零后端**：答案不离开用户设备，隐私友好
- **零构建依赖**：仅 Vite + html2canvas
- **GitHub Pages 部署**：gh-pages 分支自动同步

### 4. 模块化架构

数据与 UI 严格分离：
```
src/
├── main.js          # UI 逻辑（无硬编码数据）
├── engine.js        # 计分纯函数（无 DOM）
├── styles.css       # 样式
└── data/
    ├── questions.js # 题目数据
    └── personas.js  # 人设数据
```

## Vibecoding 流程

这是一个典型的「人机协作」项目，记录完整的开发流程：

### 阶段 1：理解需求

- 阅读原始灵感文章，提取核心框架
- 对标 SBTI，理解其交互模式和传播机制
- 明确目标用户和传播场景

### 阶段 2：设计决策

- 确定四维度模型、等级系统、人设定义
- 撰写 `docs/design-decisions.md` 作为「决策源文件」
- 所有后续开发以决策文档为准

### 阶段 3：迭代开发

**Claude 的角色**：
- 根据决策文档实现代码
- 保持数据与 UI 分离
- 跑验证（build、预览）

**人的角色**：
- 做所有设计决策
- 审阅代码，提出修改意见
- 决定何时「够用」

**协作模式**：
- 每次新对话前，Claude 先读决策文档，恢复上下文
- 决策变更时更新文档，而非口头重复
- Claude 不做未授权的设计决策

### 阶段 4：部署上线

- Vite 构建 → dist/
- git subtree 推送到 gh-pages 分支
- GitHub Pages 自动部署

### 关键经验

1. **文档先行**：决策文档是协作的「共享大脑」，避免重复沟通
2. **模块分离**：数据独立于逻辑，便于非开发者修改内容
3. **小步验证**：每次改动后跑 build，不累积错误
4. **明确边界**：Claude 不做红线决策（改维度、改人设等）
